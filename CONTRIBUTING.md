# Contributing to celily

## LLM-assisted contributions

LLM assistance (code generation, scaffolding, research) is allowed, under the
following conditions:

- **No fully automated agentic contributions.** Pull requests and issues must be
  opened and discussed by a human. Automated scripts, bots, or agents that open
  PRs without human oversight are not accepted and will be closed.

- **A human must interact on PRs and issues.** Translation software is fine. The
  person behind the contribution is the one responsible for it.

- **You must understand the code you submit.** Do not blindly paste LLM output.
  Be prepared to explain what the change does, why it's done that way, and how
  it interacts with the security boundaries described above. If you can't
  explain it, don't submit it.

- **LLM-generated code is reviewed to the same standard as human-written code.**
  Being generated by a tool does not lower the bar. If it doesn't meet the
  conventions, testing, and documentation requirements above, it doesn't land.

## Design principles

celily is built on two rules:

- Deny-by-default. Everything is forbidden unless explicitly allowed.
- Explicit is better than implicit. It's more burdensome for the user, but
  decreases risk.

## Security boundaries

- Host is trusted. The host filesystem, the user's session, the LXD/Incus
  daemon, and the celily configuration directory are all trusted. The config
  directory (`~/.config/celily/`) is validated on load (owned by the current
  user, no group/other permissions) and is permanently blocked from bind-mounts
  into containers.

- Container is untrusted. Everything inside the container is assumed to be
  untrusted. It will try to read arbitrary files, connect to arbitrary hosts,
  and exfiltrate data (whether by mistake or by purpose).

- Mount validation is the filesystem boundary. Every bind-mount source is
  validated before the container starts: must be under `$HOME`, must be in the
  exact-match allowlist, must not hit the forbidden-path list. Symlinks are
  resolved before checking. Subtrees of allowed directories are not implicitly
  allowed (explicit is better than implicit).

- The proxy is the network boundary. The only egress path for HTTP traffic is
  the per-instance mitmproxy. `HTTP_PROXY` and `HTTPS_PROXY` are injected at the
  highest priority and cannot be overridden by user config or `--env`. Tools
  that bypass the proxy hit the bridge ACL and get dropped.

## Config merging

- If you add a new config field, you must add the merge logic in the appropriate
  `merge()` function or the field will silently never take effect from profiles.

- Boolean fields that default to `true` must use `Option<bool>` with a custom
  serde default. A plain `bool` cannot distinguish "not set" from "set to
  false", so `default || profile` would block profiles from turning the feature
  off.

## Testing

- Add unit tests for new code if possible.
- Integration tests for the full container lifecycle are not yet in place. If
  your change touches the orchestration layer (`main.rs`, `context.rs`, the
  type-state machine), test it manually against a real LXD/Incus instance.

## Documentation

- Every user-facing change (new CLI flag, config field, or behaviour change)
  must update the corresponding man page under `man/`. The man pages are written
  in scdoc format.
- The README should stay current with major feature additions or changes to the
  security model.
- Man pages must build cleanly with `scdoc`. Run `make man` to verify.

## Formatting

Format Rust code with cargo fmt, and python with black.

## Commit messages

We follow [Conventional Commits](https://www.conventionalcommits.org/).
